# 증강 코딩 운영 기록

## 1. 개인 검증 단계

처음엔 바이브 코딩, “AI 전면 위임” 방식에 회의적이었음.  
결과 재현성/품질 통제 어렵다고 판단했음.

이후 Kent Beck이 정의한 증강 코딩 관점 접했음.  
핵심이 전면 위임이 아니라, **사람 의도 + 검증 루프 중심 활용**이라는 점 확인했음.

그래서 팀 적용 전에 개인 프로젝트로 약 2주 검증 진행함.

- 대상: <https://github.com/currenjin/alexandria-playground/tree/main/playground-augmented-coding>

검증 결과 요약:

- 티켓 기반 구조 적용 시 재현성 올라감
- 테스트 기반 루프 유지 시 속도/안정성 같이 확보됨
- 단순 생성형 활용보다 작업 순서/검증 루프 설계가 품질에 더 큰 영향 줌

검증 후 팀 적용 진행함.

## 2. 팀 스킬 적용

개인 검증 후 팀 스킬로 운영 범위 확장함.

현재 기본 흐름은 아래와 같음.

1) Jira 티켓 읽고 `plan.md` 생성  
2) 증강 코딩 스킬이 `plan.md` 기준으로 구현 진행  
3) 테스트 통과 기준으로 수정 반복  
4) PR 생성 후 리뷰 단계 진입

초기엔 개인 사용 중심이었음.  
효용 확인되면서 팀원 사용 확대됨.

운영 원칙은 바이브 코딩 아님.  
**테스트 기반 실행** 유지함.

관측 효과:

- 작업 안정성 향상됨
- 구현/검증 속도 개선됨
- 리뷰 이전 단계 품질 편차 감소함

## 3. 하네스 방식 도입

초기엔 이 운영을 하네스라고 명시하진 않았음.  
다만 스킬 체인/검증 루프/승인 지점 누적되면서 하네스 구조로 수렴했음.  
현재도 사람 개입은 유지 중임.

적용 중 스킬 구성:

- `jira-to-plan`
- `augmented-coding`
- `push-pr`
- `review`

위 스킬 통합한 `work` 스킬 사용 중임.

현재 운영 상태:

- 병렬 작업 처리 가능함
- 작업 분리 + 검증 루프로 안정성 유지 중임
- 사람은 도메인 정책/요구사항/가독성 중심 검토함

## 4. kancli 개발 배경

병렬 작업 수 증가하면서 운영 UI 문제 발생함.  
여러 터미널 분산 운영 방식은 집중 저하/관리 비용 증가 유발함.

그래서 단일 터미널에서 칸반 방식으로 작업 관리하는 CLI 개발 중임.

- 프로젝트: <https://github.com/currenjin/kancli>

개발 목적:

- 작업 상태 단일 화면 관리
- 컨텍스트 스위칭 비용 감소
- 병렬 작업 운영 가시성 개선
