# 증강 코딩, 바이브를 넘어서

같은 모델을 쓰는데 왜 결과는 다를까?
대부분의 팀은 여기서 개인의 프롬프트 실력을 탓한다.
나는 반대로 본다.

**문제는 개인이 아니라 시스템이다.**

증강 코딩에서 성과를 가르는 건 “누가 더 똑똑한 프롬프트를 쓰느냐”가 아니라,
팀이 AI를 어떤 하네스(harness) 안에서 굴리느냐이다.

---

## 각자도생 모드가 만드는 착시

초기에 우리도 그랬다.

- 같은 Jira 티켓
- 같은 레포
- 같은 모델

그런데 결과 품질과 속도가 들쭉날쭉했다.
누군가는 10분 만에 PR을 만들고,
누군가는 1시간 동안 할루시네이션을 정리했다.

처음엔 “숙련도 차이”라고 생각했다.
지금은 안다.

**컨텍스트와 검증 루프를 팀 단위로 설계하지 않았기 때문**이다.

---

## 병목은 코딩이 아니라 리뷰

AI 도입 후 코딩 속도는 분명 빨라졌다.
하지만 실제 병목은 더 빨리, 더 선명하게 드러났다.

- 도메인 정책 검토
- 요구사항 해석 검증
- 가독성/유지보수성 판단

코드는 빨리 나오는데,
사람의 리뷰 시간이 고정되어 있으니 전체 처리량이 거기서 막혔다.

그래서 질문이 바뀌었다.

- “어떻게 더 빨리 코드를 짤까?”
- → “어떻게 사람의 판단 시간을 고부가가치 영역에만 쓰게 할까?”

---

## 우리가 고정한 작업 규율

우리 팀은 아래 규율을 먼저 고정했다.
기술보다 운영 규칙을 먼저 맞춘 셈이다.

### 1) 시작 조건
- Jira에 task 티켓이 생성되면 시작

### 2) 브랜치 규율
- 항상 `develop`에서 분기
- 기능 브랜치 형식: `feature/RP-1234`

### 3) 검증 규율
- CodeRabbit 자동 리뷰
- 백엔드는 `/review` 스킬로 2차 검증
- 전체 테스트 통과 필수

### 4) 완료 정의(DoD)
- PR 리뷰 완료
- `develop` 머지

### 5) 막힘 처리
- 막히면 즉시 질문
- 뭉개고 진행하지 않음

이 규율은 화려하지 않다.
하지만 처리량과 품질을 동시에 유지하려면, 결국 이런 기본기가 필요하다.

---

## 하네스 관점: 코드보다 제어면

AI 시대 엔지니어의 핵심 업무는 점점 이쪽으로 이동한다.

- 입력을 구조화하고
- 실행 순서를 고정하고
- 실패를 조기에 감지하고
- 사람 개입 지점을 명확히 정의하는 것

즉,

- AI는 실행 엔진
- 사람은 의사결정자
- 하네스는 둘의 경계를 설계하는 제어면

좋은 하네스는 개인의 편차를 줄인다.
잘하는 사람의 방식이 팀 표준으로 복제된다.
이게 생산성 저점을 끌어올리는 유일한 방법이다.

---

## 사람이 반드시 해야 하는 판단

자동화 실험을 할수록 더 분명해지는 것도 있다.
사람이 끝까지 붙잡아야 하는 판단이다.

1. **도메인 정책 적합성**
2. **업무 요구사항 충족 여부**
3. **장기 유지보수 관점의 가독성/구조 판단**

이걸 자동화로 대체하려고 하면,
단기 속도는 올라가도 장기 비용이 폭발한다.

---

## 문서는 기록이 아니라 실행이어야 한다

문서는 쉽게 낡는다.
그래서 문서를 기록 시스템으로만 두면 의미가 약하다.

실행 가능한 문서가 필요하다.

- 사람이 읽으면 가이드
- 에이전트가 읽으면 실행 규칙
- CI가 읽으면 차단 조건

한 줄로 요약하면 이렇다.

**설명만 있는 규칙은 규칙이 아니다. 실패를 강제할 수 있어야 규칙이다.**

---

## 우리가 아직 풀지 못한 문제

솔직히 말하면 완성된 상태는 아니다.

- 리뷰 병목 완화 방식의 팀 내 합의
- 질문 escalation 템플릿 표준화
- `/review` 실험 결과의 정량화
- 자동화 범위와 사람 승인 경계의 재조정

하지만 방향은 명확하다.
코드 생성량이 아니라,
**의도-실행-검증 루프의 안정성**이 생산성을 만든다.

---

## 결론

증강 코딩은 도구 도입 프로젝트가 아니다.
팀 운영체계 설계 프로젝트다.

개인의 프롬프트 요령은 중요하다.
하지만 팀 성과를 바꾸는 건 개인기가 아니라 하네스다.

그래서 우리가 해야 할 일은 단순하다.

- 잘하는 사람을 더 잘하게 만드는 것보다
- 팀 전체가 망하지 않게 만드는 규율을 먼저 배포하는 것

AI 시대의 경쟁력은 코드 작성 속도만으로 생기지 않는다.
**의도를 실행 가능한 시스템으로 바꾸는 능력**에서 나온다.
