# 증강 코딩 운영 기록

증강 코딩은 코드를 덜 쓰는 요령이 아님.  
의도와 검증을 더 엄격하게 다루는 운영 방식임.

Kent Beck이 말한 관점도 같음.  
핵심은 전면 위임이 아니라, 사람이 목표/제약/판정 기준을 먼저 설계하고 AI를 그 안에서 실행시키는 것임.

---

## 1) 시작점: 왜 회의적이었는지

처음엔 바이브 코딩, AI 전면 위임 둘 다 회의적이었음.  
이유는 단순했음.

- 결과 재현성 떨어짐
- 품질 통제 어려움
- 작업자마다 결과 편차 큼

그래서 “일단 써보자”가 아니라 “검증부터 하자”로 접근했음.

개인 프로젝트에서 약 2주 검증함.

- <https://github.com/currenjin/alexandria-playground/tree/main/playground-augmented-coding>

검증에서 확인한 사실:

- 티켓 기반 구조로 가면 재현성 올라감
- 테스트 루프 강제하면 속도/안정성 같이 확보됨
- 프롬프트 감각보다 작업 순서/검증 구조가 결과를 더 크게 좌우함

이 결과를 보고 팀 적용 시작함.

---

## 2) 팀 적용: 개인기에서 규율로

현재 기본 플로우는 고정해 둠.

1. Jira 티켓 읽고 `plan.md` 생성
2. `plan.md` 기준으로 구현 진행
3. 테스트 통과 기준으로 수정 반복
4. PR 생성 후 리뷰

초기엔 개인 사용 중심이었음.  
효용 확인되면서 팀원 사용 확대됨.

중요한 점은 여기임.

- 바이브 코딩으로 운영하지 않았음
- 테스트 기반 루프를 기본 규율로 둠

관측된 변화:

- 작업 안정성 상승
- 구현/검증 속도 개선
- 리뷰 전 단계 품질 편차 감소

---

## 3) 하네스 방식으로 수렴

처음부터 “하네스”라는 용어를 쓰고 시작한 건 아님.  
근데 운영을 쌓다 보니 구조가 하네스로 수렴했음.

현재 스킬 체인:

- `jira-to-plan`
- `augmented-coding`
- `push-pr`
- `review`

위 스킬을 `work` 스킬로 통합해서 사용 중임.

이 구조의 의미는 명확함.

- 사람: 목표/제약/완료 조건 정의
- AI: 구현/수정/반복 실행
- 리뷰: 자동 + 사람 판단 결합

즉, 팀 생산성의 저점을 개인 역량이 아니라 실행 규율로 끌어올리는 방식임.

---

## 4) 사람 개입은 어디에 남겨두는지

전면 자동화는 목표가 아님.  
사람이 반드시 판단해야 할 영역은 남겨둠.

- 도메인 정책 적합성
- 업무 요구사항 충족 여부
- 장기 유지보수 관점의 가독성/구조 판단

자동화는 실행량을 늘리는 데 쓰고,  
사람은 고비용 판단에 집중시키는 게 현재 운영 원칙임.

---

## 5) kancli를 만드는 이유

병렬 작업이 늘어나면서 운영 UI가 병목이 됨.  
터미널 여러 개 분산 운영은 집중력/관리 비용 측면에서 비효율적이었음.

그래서 단일 터미널에서 칸반 방식으로 관리하는 CLI를 개발 중임.

- <https://github.com/currenjin/kancli>

목표:

- 작업 상태를 한 화면에서 관리
- 컨텍스트 스위칭 비용 감소
- 병렬 작업 운영 가시성 개선

---

## 6) 현재 결론

지금까지의 결론은 명확함.

- 증강 코딩 성과는 모델 선택보다 운영 설계에서 갈림
- 개인 프롬프트 실력보다 팀 하네스가 결과를 안정화함
- “많이 생성”보다 “의도-검증 루프” 품질이 더 중요함

즉, 증강 코딩은 도구 도입 이슈가 아니라 팀 운영체계 설계 이슈임.
