# 증강 코딩 운영 기록

## 1) 시작점: 처음엔 회의적이었음

처음엔 바이브 코딩, “AI한테 전부 맡기는 방식” 둘 다 회의적이었음.  
이유는 단순했음. 재현성 떨어지고, 품질 통제가 안 된다고 봤기 때문.

그러다 Kent Beck이 말한 증강 코딩 관점을 보게 됐음.  
핵심은 전면 위임이 아니라 **사람 의도 + 검증 루프 중심 운영**이라는 점이었음.

그래서 바로 팀에 넣지 않고, 개인 프로젝트에서 먼저 2주 검증함.

- <https://github.com/currenjin/alexandria-playground/tree/main/playground-augmented-coding>

확인한 내용:

- 티켓 기반으로 작업하면 재현성 올라감
- 테스트 루프를 강제하면 속도/안정성 같이 확보됨
- “프롬프트 잘 쓰기”보다 “작업 순서 + 검증 구조”가 결과를 더 크게 좌우함

이 정도 확인하고 팀 적용 시작함.

## 2) 팀 적용 방식

현재 기본 흐름은 고정해 둠.

1. Jira 티켓 읽고 `plan.md` 생성
2. 증강 코딩 스킬이 `plan.md` 기준으로 구현
3. 테스트 통과 기준으로 수정 반복
4. PR 생성 후 리뷰

초기엔 나만 썼는데, 효과가 보이면서 팀원도 쓰기 시작함.

중요한 건 여기서 바이브 코딩으로 안 갔다는 점임.  
항상 테스트 기준으로 밀어붙였음.

관측된 변화:

- 작업 안정성 올라감
- 구현/검증 속도 빨라짐
- 리뷰 전에 품질 편차 줄어듦

## 3) 하네스 방식으로 수렴

처음부터 “하네스”라고 이름 붙이고 시작한 건 아님.  
근데 운영하다 보니 자연스럽게 하네스 구조가 됐음.

현재 스킬 구성:

- `jira-to-plan`
- `augmented-coding`
- `push-pr`
- `review`

그리고 이걸 `work` 스킬로 묶어서 사용 중.

핵심은 이거였음:

- 병렬 작업 처리 가능
- 검증 루프 유지한 상태로 속도 확보
- 사람은 도메인/요구사항/가독성 판단에 집중

## 4) kancli를 만드는 이유

병렬 작업 늘어나니까 제일 먼저 터진 게 UI였음.  
터미널 여러 개 켜서 추적하는 방식은 집중력 박살 + 관리 비용 증가.

그래서 한 터미널에서 칸반으로 관리하는 CLI를 직접 만들고 있음.

- <https://github.com/currenjin/kancli>

목표는 명확함:

- 작업 상태를 한 화면에서 관리
- 컨텍스트 스위칭 줄이기
- 병렬 작업 운영 가시성 높이기
