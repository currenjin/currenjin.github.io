# 증강 코딩 (Augmented Coding)

이 문서는 “AI를 코딩 보조 도구로 쓰는 것”을 넘어, 팀 생산성을 높이기 위한 **운영체계**로서 증강 코딩을 정리한 글이다.

---

## 1. 왜 증강 코딩을 하는가

많은 팀이 같은 LLM, 같은 IDE를 쓰지만 결과는 크게 다르다.
차이는 보통 모델 자체보다 **맥락 설계, 검증 루프, 리뷰 방식**에서 발생한다.

우리 팀의 문제도 비슷했다.

- 코딩 속도는 빨라졌는데 리뷰가 병목이 됨
- 사람마다 AI 활용법 편차가 큼
- 잘못된 기본값(브랜치, 버전, 규칙 누락)으로 재작업 발생

증강 코딩의 목적은 단순하다.

> 개인의 요령이 아니라, 팀 전체가 재현 가능한 방식으로 빠르게 개발하도록 만드는 것.

---

## 2. 우리 팀의 현재 기본 플로우

### 시작
- Jira에 task 티켓 생성되면 개발 시작
- 기본 브랜치: `develop`
- 기능 브랜치: `feature/RP-1234`

### 구현/검증
- AI 기반 구현 + 테스트 보강
- 전체 테스트 통과 필수

### 리뷰
- CodeRabbit 자동 리뷰
- `/review` 스킬로 추가 검증
- 핵심 판단(도메인 정책/요구사항/가독성)은 사람 검토

### 완료 정의(DoD)
- PR 리뷰 완료
- `develop` 머지

---

## 3. 핵심 관점 전환: “코드 작성”에서 “하네스 설계”로

증강 코딩을 잘하기 위해서는 프롬프트 문장력보다 **하네스(Harness)**가 중요하다.

하네스는 아래를 의미한다.

- 어떤 입력으로 시작하는지
- 어떤 순서로 실행하는지
- 어디서 실패를 감지하는지
- 언제 사람에게 질문하는지
- 무엇을 완료로 판단하는지

즉, AI는 엔진이고 하네스는 제어장치다.

---

## 4. 팀 적용 시 필수 구성요소

## 4-1. Executable SSOT

문서가 단순 기록으로 끝나면 금방 낡는다.
따라서 규칙은 다음 두 형태로 함께 관리해야 한다.

1. 사람이 읽는 짧은 가이드
2. 시스템이 강제하는 실행 규칙(CI/린트/테스트)

원칙:
- 규칙을 설명만 하지 말고, 실패시키는 조건으로 올린다.

## 4-2. Preflight 체크

개발 시작 전에 자동으로 확인한다.

- base branch가 `develop`인지
- 브랜치 네이밍 규칙 준수 여부
- 규칙 문서/스킬 참조 가능 여부
- 테스트 환경 실행 가능 여부

잘못된 상태에서 코딩을 시작하지 않는 것이 핵심.

## 4-3. 리뷰 이원화

- AI 리뷰: 반복적/기계적 검증
- 사람 리뷰: 도메인 의도/트레이드오프 판단

사람은 “찾기”보다 “판단”에 집중해야 한다.

## 4-4. 질문 포맷 표준화

막히면 즉시 질문하되, 질문 형식은 고정한다.

- 문제 요약
- 선택지 A/B
- 추천안
- 영향 범위

이렇게 해야 답변 속도와 품질이 올라간다.

---

## 5. 실전 운영 템플릿

### 5-1. 작업 시작 템플릿

```text
[START]
- Ticket: RP-1234
- Base: develop
- Branch: feature/RP-1234
- Goal:
- Acceptance Criteria:
```

### 5-2. 리뷰 템플릿

```text
[REVIEW]
- CodeRabbit 요약:
- /review 결과:
- 사람 검토 포인트(도메인/요구사항/가독성):
- 조치 여부:
```

### 5-3. 질문 템플릿

```text
[QUESTION]
- Ticket:
- Problem:
- Option A:
- Option B:
- Recommended:
- Impact:
```

---

## 6. 흔한 실패 패턴

1. 규칙이 문서에만 있고 실행 강제가 없음
2. 테스트 통과보다 “빨리 PR”이 우선됨
3. AI 리뷰 결과를 사람이 다시 읽지 않음
4. 질문이 자유서술이라 의사결정이 느려짐

증강 코딩은 자동화 비율보다 **피드백 루프 품질**이 더 중요하다.

## 7. 도입 로드맵 (2주)

### Week 1
- 팀 공통 시작/리뷰/질문 템플릿 확정
- Preflight 체크 항목 확정
- 리뷰 책임 분리(자동 vs 사람)

### Week 2
- 반복 실수 3개를 하드 가드레일로 승격
- 리뷰 리드타임 측정 시작
- 회고에서 규칙 업데이트

추천 KPI:
- PR 리드타임
- 리뷰 재작업 횟수
- 머지 후 핫픽스 건수

## 결론

증강 코딩은 “AI를 쓰는 법”이 아니라 “팀이 일하는 법”의 문제다.
핵심은 사람을 제거하는 것이 아니라, 사람이 더 중요한 판단에 집중하도록 시스템을 설계하는 것이다.

- AI는 실행을 담당하고
- 사람은 의도를 결정하며
- 하네스는 둘 사이를 안정적으로 연결한다.
