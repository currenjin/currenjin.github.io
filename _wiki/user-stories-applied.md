# 사용자 스토리 (마이클 콘)

대상 도서: **User Stories Applied (Mike Cohn)**  
목표: "요구사항 문서"가 아니라 "대화 가능한 약속"으로 사용자 스토리를 쓰고 운영하기.

---

## 1) 사용자 스토리의 본질

사용자 스토리는 상세 명세서가 아니다.

- 스토리 = **대화의 트리거**
- 구현 전부를 적는 문서가 아니라, 팀이 함께 구체화하기 위한 약속
- 핵심: **Card / Conversation / Confirmation (3C)**

### 3C

- **Card**: 짧은 스토리 문장(기억 장치)
- **Conversation**: PO/개발/QA가 의도·범위·예외를 대화로 정리
- **Confirmation**: 수용 기준(acceptance criteria)으로 완료 조건 확정

---

## 2) 스토리 템플릿

가장 기본적인 형태:

> As a [사용자 유형], I want [원하는 것], so that [얻는 가치].

예시:

> As a 구매자, I want 주문 내역을 기간별로 조회, so that 월별 지출을 확인할 수 있다.

주의:
- "개발자가 ~한다" 형태로 쓰지 않는다.
- 구현 방식(DB, API 내부 구조)보다 **사용자 가치**가 먼저다.

---

## 3) 좋은 스토리 기준 (INVEST)

- **I**ndependent: 가능한 독립적으로
- **N**egotiable: 협상 가능(고정 명세 아님)
- **V**aluable: 사용자/비즈니스 가치 존재
- **E**stimable: 대략 추정 가능
- **S**mall: 한 이터레이션에서 끝낼 수 있을 정도로 작게
- **T**estable: 테스트/검증 가능

실무 체크:
- 스토리에 가치 문장이 없으면 재작성
- 테스트 기준이 없다면 아직 준비 안 된 스토리

---

## 4) 수용 기준(AC) 쓰는 법

스토리 자체는 짧게, 완료 조건은 명확히.

### 좋은 AC 원칙

- 관찰 가능해야 함
- 모호한 단어("빠르게", "잘") 금지
- 정상/예외/경계값 포함

예시:

- Given 사용자가 로그인 상태일 때
- When 주문 목록에서 2026-03 기간을 선택하면
- Then 해당 월 주문만 최신순으로 1초 이내 표시된다.

---

## 5) 스토리 쪼개기(분할) 전략

큰 스토리는 바로 구현하지 않고 분할한다.

대표 분할 방식:

1. **워크플로우 단계 분리**: 조회 → 생성 → 수정 → 취소
2. **비즈니스 규칙 분리**: 기본 규칙 먼저, 예외 규칙 나중
3. **데이터 범위 분리**: 핵심 필드 우선, 부가 필드 후속
4. **인터페이스 분리**: API 우선, UI 후속(또는 반대)
5. **단순/복잡 케이스 분리**: happy path 먼저

원칙: "수직 절단"(사용자 가치가 끝까지 전달되는 작은 단위) 선호.

---

## 6) 에픽/테마/스토리 관계

- **Theme**: 상위 목적/방향 (예: 결제 경험 개선)
- **Epic**: 큰 기능 묶음 (예: 결제수단 관리)
- **Story**: 실제 구현 가능한 단위

운영 팁:
- 초기에 Epic으로 두고, 스프린트 진입 전 Story로 분해
- 스프린트 중간에 너무 큰 스토리 발견 시 즉시 재분할

---

## 7) 추정(Estimation) 핵심

이 책의 포인트는 "정답 시간"보다 "상대 비교"다.

- 스토리 포인트로 상대적 크기 추정
- 팀 내 기준 스토리를 앵커로 고정
- 불확실성이 큰 항목은 스파이크(조사)로 위험 줄이기

주의:
- 추정치 = 약속 시간이 아님
- 추정은 학습 데이터가 쌓일수록 정확해진다.

---

## 8) 계획/우선순위 운영

우선순위는 단순 중요도 1축이 아니다.

함께 보는 요소:
- 비즈니스 가치
- 위험 감소 효과
- 학습 가치(빠른 피드백)
- 구현 비용

추천:
- 초기에 위험 큰 스토리를 앞에 둬서 불확실성 제거
- "나중에 알면 늦는 것"을 먼저 검증

---

## 9) 안 좋은 스토리 신호

- 기술 태스크만 적혀 있음 (사용자 없음)
- 너무 큼 (한 스프린트 내 종료 불가)
- 완료 조건 없음
- 팀원마다 해석이 다름
- 의존성이 과도해 단독 진행 불가

대응:
- 가치 문장 추가
- AC 작성
- 분할
- 의존성 제거 순서 재설계

---

## 10) 실무 적용 템플릿

### 스토리 카드 템플릿

```text
[Story]
As a [role]
I want [goal]
so that [value]

[Acceptance Criteria]
1)
2)
3)

[Notes]
- 범위 제외:
- 의존성:
- 리스크:
```

### 스프린트 진입 체크리스트

- [ ] INVEST 최소 충족
- [ ] AC 3개 이상
- [ ] 범위 제외 명시
- [ ] 테스트 가능
- [ ] 팀 공통 이해 확인

---

## 11) 내 업무에 바로 적용 액션

1. 이슈 생성 시 "As/I want/so that" 강제
2. PR 본문에 AC 매핑(각 AC를 어떻게 검증했는지)
3. 큰 이슈는 먼저 Epic으로 두고 Story로 재분해
4. 스프린트 시작 전 "분할 품질 리뷰" 15분 고정
5. 완료 정의(DoD)와 AC를 분리해 관리

---

## 요약

마이클 콘의 사용자 스토리는 한 줄 양식이 핵심이 아니다.
핵심은 **대화와 검증이 가능한 작은 가치 단위**로 요구를 다루는 운영 방식이다.

- 스토리는 짧게
- 대화는 충분히
- 확인(AC)은 구체적으로
- 구현은 작고 독립적으로

이 네 가지가 지켜지면 스프린트 예측 가능성과 리뷰 품질이 동시에 오른다.
