# Kotlin vs Java 트레이드오프 정리

이 문서는 "어떤 언어가 더 좋다"가 아니라,
**팀/제품/운영 맥락에서 어떤 선택이 더 유리한지**를 정리한다.

---

## 1) 한 줄 요약

- **Java**: 안정성, 생태계, 조직 표준화에 강함
- **Kotlin**: 생산성, 표현력, null-safety에 강함

결론은 언어 우열이 아니라 **비용 구조가 다르다**는 점이다.

---

## 2) 핵심 차이 (실무 체감)

## 2-1. Null 안정성

- Java: nullable/nonnull을 타입 시스템에서 강제하지 않음
- Kotlin: nullable 타입(`String?`)을 언어 레벨에서 강제

영향:

- Kotlin은 NPE를 컴파일 단계에서 많이 줄인다.
- Java는 팀 컨벤션(어노테이션/리뷰)에 더 의존한다.

## 2-2. 문법/표현력

Kotlin은 같은 기능을 더 짧게 표현하는 경우가 많다.

- data class
- extension function
- sealed class
- scope function

영향:

- Kotlin은 개발 속도가 빨라질 수 있음
- 반대로 팀 숙련도가 낮으면 읽기 난이도가 오를 수 있음

## 2-3. 호환성과 생태계

둘 다 JVM 위에서 동작하고 상호 호출 가능하다.

- Java: 레거시 라이브러리/문서/사례가 압도적으로 많음
- Kotlin: JVM + Android + 멀티플랫폼 확장성

## 2-4. 빌드/컴파일 체감

일반적으로 Kotlin이 컴파일 체감이 더 무거운 경우가 있다.
(프로젝트 규모/Gradle 설정/증분 컴파일 상태에 따라 다름)

영향:

- 대형 모노레포에선 빌드 최적화가 중요
- 작은 서비스는 생산성 이점이 더 크게 느껴짐

## 2-5. 팀 운영 관점

- Java는 팀 내 편차가 상대적으로 작고 온보딩이 쉬운 편
- Kotlin은 숙련도 격차에 따라 코드 스타일 편차가 커질 수 있음

---

## 3) Kotlin 장점/단점

### 장점

- null-safety로 런타임 오류 감소
- 보일러플레이트 감소 (data class 등)
- 코루틴 기반 비동기 코드 가독성 우수
- Spring Boot와 궁합이 좋고 실무 사례 충분

### 단점

- 고급 문법 남용 시 코드 이해도 저하
- Java-only 팀에서는 학습비용 발생
- 일부 라이브러리에서 Java 관점 API가 더 자연스러운 경우 있음

---

## 4) Java 장점/단점

### 장점

- 성숙한 생태계와 방대한 참고자료
- 대규모 조직 표준화/운영 안정성
- 채용/온보딩 수급 측면에서 유리한 경우 많음
- 성능 튜닝, 도구 지원, 관측성 경험이 풍부

### 단점

- 보일러플레이트가 상대적으로 많음
- nullable 안전성이 언어 차원 강제가 아님
- 기능 확장 시 코드량 증가로 생산성 저하 가능

---

## 5) 상황별 선택 가이드

## 5-1. Java가 더 유리한 경우

- 대형 조직/장기 유지보수 중심
- 팀의 Java 숙련도가 높고 Kotlin 경험이 낮음
- 레거시 Java 자산 비중이 매우 큼
- 표준화/일관성 우선

## 5-2. Kotlin이 더 유리한 경우

- 신규 서비스 중심, 빠른 반복 개발 필요
- null 안정성과 개발 생산성이 중요
- 팀 내 Kotlin 숙련도 확보
- 도메인 모델 표현력과 코드 간결성이 중요한 프로젝트

## 5-3. 혼합 전략이 유효한 경우

- 기존 Java 서비스 유지 + 신규 모듈 Kotlin 도입
- 경계가 명확한 패키지/모듈 단위로 점진 전환
- 공통 룰(코드 스타일, 코루틴 사용 규칙) 먼저 합의

---

## 6) 트레이드오프 표 (요약)

- 개발 속도
  - Kotlin 우세(보일러플레이트 감소)
- 런타임 안정성(Null)
  - Kotlin 우세(타입 시스템)
- 조직 표준화
  - Java 우세
- 온보딩 난이도
  - Java 우세(평균적으로)
- 표현력/간결성
  - Kotlin 우세
- 빌드/툴링 성숙도
  - Java 우세(여전히)

---

## 7) 팀 적용 시 주의할 점

1. 언어보다 **규칙**이 먼저
- 예: 코루틴 사용 범위, nullable 규칙, 예외 처리 규칙

2. Kotlin 도입 시 코드 스타일 가드 필요
- 과도한 scope function 체인 금지
- 난해한 extension 남용 금지

3. 혼합 코드베이스면 경계 명확화
- Java↔Kotlin interop 구간 정책 정리

4. 성능/빌드는 측정 기반으로 판단
- 체감이 아니라 빌드 시간/배포 리드타임으로 비교

---

## 8) 추천 의사결정 프레임

언어 선택 전에 아래 4가지를 점검하면 흔들림이 줄어든다.

- 팀 숙련도: 지금 당장 생산성 내는가?
- 제품 속도: 실험/변경이 잦은가?
- 운영 안정성: 장애 비용이 얼마나 큰가?
- 채용/유지보수: 2년 뒤 운영이 쉬운가?

---

## 9) 결론

- Kotlin vs Java는 우열 문제가 아니라 **운영 전략 문제**다.
- 안정성과 표준화를 우선하면 Java가 유리한 경우가 많다.
- 생산성과 표현력을 우선하면 Kotlin이 강력하다.
- 대부분 팀의 현실 해법은 "한 번에 교체"가 아니라 **점진적 혼합 전략**이다.
